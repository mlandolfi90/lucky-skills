## [GAP-003] El HEAD del branch tiene imagen y CI verde pero el deploy nunca sale (pushes encadenados + gate docs-only)

- **TIPO:** GAP (hueco del pipeline: la combinación concurrency + gate deja un estado sin dueño)
- **SÍNTOMA (lo observable, NO la causa):** Pusheaste un fix y esperás el deploy; pasan >10 min y
  el entorno sigue sirviendo el sha viejo. En Actions: el run del fix figura `cancelled`, el run
  del HEAD figura `success`… pero con el job `deploy` `skipped`.
- **CAUSA-RAÍZ (1 línea):** la concurrency del workflow cancela los runs intermedios cuando
  pusheás en cadena, y si el ÚLTIMO commit es docs-only el gate "Detect deployable change" no
  deploya — el run que sí habría deployado (el del fix) murió cancelado: build hay, deploy no.
- **ACCIÓN (pasos, máx 7):**
  1. Confirmá el estado: sha vivo (`/api/version` o equivalente) ≠ HEAD, y en Actions el último
     run success con `deploy: skipped` + los anteriores `cancelled`.
  2. Disparo manual: `workflow_dispatch` del workflow sobre el branch — el gate deploya SIEMPRE
     en dispatch. No hace falta commit vacío.
  3. Verificá el sha vivo tras el redeploy (durante el switch puede haber 503/CSRF transitorios —
     es el reinicio, no un fallo nuevo).
- **ANTI-ACCIÓN (el camino muerto — evita re-derivar):** NO push de commit vacío/dummy para
  "despertar" el deploy (ensucia historia y puede volver a cancelarte si seguís pusheando); NO
  re-run del run cancelado (re-corre sobre el sha viejo, no sobre HEAD); NO tocar Coolify a mano
  (el disparador canónico es el job deploy del CI).
- **PREVENCIÓN (cómo evitar reincidencia):** si vas a encadenar un commit docs-only detrás de un
  fix, dale al fix tiempo de deployar o metelos en el MISMO push; tras cualquier tanda de pushes
  encadenados, chequear que el último run tenga `deploy` ejecutado (no skipped).
