## [GAP-007] Un `replace_all` pisa una línea que ACABÁS de agregar y que matchea el patrón → `null` en runtime

- **TIPO:** GAP
- **SÍNTOMA (lo observable, NO la causa):** tras un reemplazo masivo (`replace_all`/sed de `foo(` → `bar(`), una función explota en runtime con `TypeError: Cannot read properties of null` (ej. `null.appendChild`) en una línea que VOS mismo agregaste momentos antes del reemplazo. Los tests unitarios pasan (no ejercitan ese path).
- **CAUSA-RAÍZ (1 línea):** el `replace_all` no distingue tu línea nueva de las viejas — si tu línea (la EXCEPCIÓN al patrón) matchea, la pisa también. Caso real: agregué `document.body.appendChild(hostEl)` y después corrí `replace_all "document.body.appendChild(" → "shadowRoot.appendChild("` → mi host quedó `shadowRoot.appendChild(hostEl)` con `shadowRoot` aún null.
- **ACCIÓN (pasos):**
  1. **ORDEN:** corré el `replace_all` ANTES de agregar la línea-excepción (así tu línea nueva no existe cuando el replace barre).
  2. Si no podés reordenar: tras el `replace_all`, RELÉ a mano la región que editaste vos.
  3. Verificá el invariante: `grep` que la línea-excepción quedó con el target correcto (`grep -c "body.appendChild(host)"` = 1).
- **ANTI-ACCIÓN (camino muerto):** NO confíes en que los tests unitarios lo cazan — un footgun en un handler DOM (activate/init) no se ejercita en `node --test`; lo caza el runtime o un verificador que razone el flujo.
- **PREVENCIÓN:** para renames/moves masivos, separá el barrido mecánico de las líneas-excepción; verificador fresco que razone el path de init (no solo compile).
