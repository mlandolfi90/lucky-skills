## [GAP-002] Agregás un workflow de CI con `schedule:` (cron) y jamás corre — 0 runs, sin error visible

- **TIPO:** GAP (hueco de conocimiento del CI) + sabor FALSO-VERDE (creés tener cobertura periódica y es teatro)
- **SÍNTOMA (lo observable, NO la causa):** un workflow recién agregado con `on: schedule` nunca
  dispara — 0 runs en Actions, sin error ni warning, silencio total. Típico en canaries / monitores
  periódicos que "quedaron listos" y nadie nota que no corren.
- **CAUSA-RAÍZ (1 línea):** los `schedule` de GitHub Actions corren SOLO desde la RAMA DEFAULT del
  repo; si el repo opera dev-only (o el workflow vive solo en un branch), el cron queda INERTE.
- **ACCIÓN (pasos, máx 7):**
  1. Antes de agregar un cron a Actions: chequeá cuál es la rama default y si el workflow va a vivir
     (actualizado) ahí.
  2. Repo dev-only → el periódico NO va en Actions: usá un scheduler independiente de la rama
     (watchdog del host, monitor externo de uptime).
  3. Si ya lo agregaste en una rama no-default: RETIRALO (es teatro) y documentá el porqué en el
     postmortem/ledger — evita que el próximo lo "restaure".
- **ANTI-ACCIÓN (el camino muerto — evita re-derivar):** dejar el workflow con cron en `dev` "por las
  dudas" — jamás va a correr y da sensación falsa de cobertura (peor que no tenerlo: nadie busca el
  monitor que "ya existe"). Tampoco lo cures moviéndolo a `main` si `main` no se promueve: quedaría
  corriendo sobre código viejo.
- **PREVENCIÓN:** al revisar un diff que agrega `schedule:`, verificar rama default vs dónde vive el
  workflow; para repos dev-only, los canaries van SIEMPRE fuera de Actions. OJO: aún en la default,
  GitHub AUTO-DESACTIVA el `schedule` tras ~60 días sin actividad del repo — el mismo teatro por otra puerta.
